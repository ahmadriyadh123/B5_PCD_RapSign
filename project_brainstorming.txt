Bagian 1 — Ideation & Problem–Solution Fit
1. Latar Belakang Masalah
Dalam kegiatan organisasi, terutama pada rapat formal, sering digunakan dokumen digital seperti berita acara, surat keputusan, atau lembar persetujuan yang memerlukan tanda tangan sebagai bentuk validasi. Namun, penggunaan tanda tangan digital berbentuk gambar masih memiliki celah pemalsuan, misalnya:
Menyalin tanda tangan dari dokumen lain.
Mengedit tanda tangan menggunakan aplikasi desain.
Menggunakan tanda tangan orang lain tanpa izin.
Akibatnya, keaslian dokumen menjadi sulit dipastikan dan dapat menimbulkan masalah administratif maupun penyalahgunaan identitas.
Karena itu, diperlukan sistem yang mampu melakukan verifikasi keaslian tanda tangan secara otomatis menggunakan pendekatan Pengolahan Citra Digital (PCD) dan Machine Learning (ML).
2. Domain dan Fokus Permasalahan
Domain
Bidang yang dipilih adalah:
Edukasi
Organisasi

“Bagaimana memverifikasi keaslian tanda tangan digital pada dokumen rapat organisasi untuk mencegah pemalsuan?”
Masalah ini dipilih karena:
Proses verifikasi manual memerlukan waktu.
Pengurus organisasi sering tidak dapat memastikan tanda tangan asli atau palsu.
Dokumen digital semakin sering digunakan dibanding dokumen fisik.

Bagian 2 — Deep Dive PCD & ML Pipeline
1. Gambaran Umum Pipeline
Pada tahap ini, sistem mulai masuk ke proses teknis pengolahan citra digital (PCD) dan Machine Learning (ML). Fokus utama bagian ini adalah bagaimana gambar tanda tangan diproses sebelum dikirim ke model ML agar hasil prediksi lebih akurat dan konsisten.
Pipeline sistem secara umum:
Input Citra
    ↓
Preprocessing PCD
    ↓
Ekstraksi Fitur
    ↓
ML Inference
    ↓
Similarity Scoring
    ↓
Mapping Koordinat
    ↓
UI Overlay
2. Tahapan PCD Sebelum Inference
Sebelum model ML melakukan prediksi, citra tanda tangan harus diproses terlebih dahulu agar memiliki format yang seragam.
3. Konversi ke Grayscale
Tujuan
Mengubah citra berwarna menjadi citra abu-abu untuk:
Mengurangi kompleksitas data.
Memfokuskan model pada bentuk tanda tangan.
Mengurangi pengaruh warna latar belakang.
Mengapa Grayscale Penting?
Model tidak membutuhkan informasi warna karena:
Tanda tangan lebih bergantung pada pola garis.
Warna tinta tidak terlalu memengaruhi identitas tanda tangan.
Dengan grayscale:
Dimensi data lebih kecil.
Proses inference lebih cepat.
Konsumsi memori lebih rendah.
Konsep Perubahan
Dari:
RGB (3 channel)
Menjadi:
Grayscale (1 channel)
Dampak ke Sistem
Keuntungan:
Efisiensi komputasi meningkat.
Training lebih stabil.
Noise warna berkurang.
4. Normalisasi Piksel
Tujuan
Mengubah rentang nilai piksel dari:
0 – 255
Menjadi:
0.0 – 1.0
Rumus Normalisasi
x_{normalized} = \frac{x}{255}
Mengapa Diperlukan?
Karena model neural network bekerja lebih stabil jika:
Data berada pada rentang kecil.
Variasi numerik tidak terlalu besar.
Dampak Positif
Mengurangi exploding gradient.
Mempercepat konvergensi model.
Prediksi menjadi lebih konsisten.
5. Center Crop
Tujuan
Menjaga rasio dan posisi tanda tangan agar konsisten.
Masalah yang sering terjadi:
Tanda tangan terlalu kecil.
Tanda tangan bergeser ke samping.
Banyak area kosong.
Center crop membantu mengambil area inti tanda tangan.
Mekanisme
Sistem:
Menentukan area tengah gambar.
Memotong area yang tidak penting.
Menjaga fokus pada objek utama.
Dampak ke ML
Tanpa crop:
Model sulit fokus.
Banyak noise.
Dengan crop:
Fitur lebih jelas.
Akurasi meningkat.
6. Resize ke Ukuran Input Model
Setelah crop, gambar diubah ukurannya agar sesuai dengan ukuran input model.
Contoh:
320 × 320
atau
224 × 224
Mengapa Harus Konsisten?
Model ML membutuhkan dimensi tetap karena:
Struktur tensor harus sama.
Layer CNN memiliki ukuran input tertentu.
7. Ekstraksi Fitur Tekstur dan Kontur
Setelah preprocessing selesai, sistem mulai mengambil karakteristik tanda tangan.
a. Fitur Tekstur
Digunakan untuk menganalisis:
Ketebalan tinta.
Kepadatan goresan.
Pola distribusi piksel.
Contoh fitur:
Local Binary Pattern (LBP)
Histogram tekstur
b. Fitur Kontur
Digunakan untuk:
Mengenali bentuk unik.
Mengidentifikasi pola lengkungan.
Mendeteksi struktur garis.
Teknik yang digunakan:
Edge Detection
Contour Extraction
8. Main Isolate vs Background Isolate
Pada Flutter, proses berat tidak boleh dijalankan di UI thread karena dapat menyebabkan lag.
Karena itu sistem membagi proses menjadi dua isolate.
9. Main Isolate
Fungsi
Menangani:
Rendering UI.
Menampilkan kamera.
Overlay bounding box.
Interaksi pengguna.
Tanggung Jawab
Main isolate fokus pada:
Responsivitas aplikasi.
Tampilan real-time.
Animasi dan feedback visual.
Komponen yang Berjalan
Contoh:
Camera preview
Status indicator
Vibrasi
Overlay hasil deteksi
10. Background Isolate
Fungsi
Menjalankan proses berat seperti:
Konversi gambar.
Preprocessing tensor.
Ekstraksi fitur.
ML inference.
Mengapa Perlu Background Isolate?
Karena proses image processing cukup berat dan dapat menyebabkan:
Frame drop.
UI freeze.
Kamera tersendat.
Dengan isolate:
UI tetap lancar.
Inference berjalan paralel.

12. Konversi CameraImage ke Tensor
Model ML tidak dapat langsung membaca gambar kamera.
Karena itu gambar harus diubah menjadi tensor.
Bentuk Tensor
Contoh:
[1, 320, 320, 1]
Penjelasan:
1 = batch size
320 × 320 = ukuran gambar
1 = grayscale channel
13. ML Inference
Proses
Tensor yang telah diproses dikirim ke model ML untuk:
Membandingkan pola tanda tangan.
Menghasilkan similarity score.
Output Model
Contoh:
0.92
Artinya:
Tingkat kemiripan 92%.
14. Threshold Similarity
Sistem menentukan valid atau tidak berdasarkan threshold.
Contoh:
Threshold = 0.85
Artinya:
≥ 0.85 → valid
< 0.85 → invalid
15. Mapping Koordinat Hasil Deteksi
Masalah umum:
Ukuran model berbeda dengan ukuran layar HP.
Contoh:
Model: 320×320
Layar HP: 1080×2400
Karena itu koordinat hasil model harus ditransformasikan.
16. Scaling Factor Ratio
Sistem menghitung rasio skala agar bounding box tepat posisi.
Rumus Scaling
scale_x = \frac{screen\ width}{model\ width}
scale_y = \frac{screen\ height}{model\ height}
Contoh
Jika:
Model width = 320
Screen width = 1080
Maka:
1080 / 320 = 3.375
Semua koordinat bounding box dikalikan faktor tersebut.
17. Overlay Bounding Box
Setelah scaling selesai:
Posisi tanda tangan ditampilkan di layar.
Bounding box mengikuti hasil deteksi.
Jenis warna:
Hijau → valid
Merah → tidak valid
18. Tantangan Teknis
a. Noise Dokumen
Masalah:
Bayangan kamera.
Kertas kusut.
Cahaya tidak merata.
Solusi:
Grayscale.
Crop.
Thresholding.
b. Variasi Bentuk Tanda Tangan
Masalah:
Tekanan pena berbeda.
Ukuran berubah.
Sudut penulisan berubah.
Solusi:
Training dataset lebih banyak.
Normalisasi ukuran.
c. Latency
Masalah:
Inference lambat.
Solusi:
Background isolate.
Resize input kecil.
Tensor optimization.

20. Kesimpulan Bagian 2
Pada tahap Deep Dive PCD & ML Pipeline, sistem dirancang agar mampu melakukan preprocessing citra secara efisien sebelum inference dilakukan. Tahapan seperti grayscale, normalisasi, center crop, dan resize membantu menghasilkan input yang konsisten bagi model ML.
Selain itu, penggunaan background isolate memungkinkan proses image processing berjalan tanpa mengganggu performa UI. Hasil inference kemudian dipetakan kembali ke ukuran layar menggunakan scaling factor sehingga overlay bounding box dapat tampil secara akurat pada perangkat pengguna.

Bagian 3 — Arsitektur & Robustness
1. Gambaran Umum Arsitektur Sistem
Pada bagian ini, fokus utama bukan lagi hanya pada proses pengolahan citra dan ML, tetapi pada bagaimana sistem dirancang agar:
Stabil,
Mudah dikembangkan,
Aman terhadap lifecycle aplikasi,
Mudah dipelihara,
Dan memiliki struktur kode yang scalable.
Karena aplikasi melibatkan:
kamera,
machine learning,
database lokal,
sinkronisasi cloud,
serta UI real-time,
maka arsitektur yang baik menjadi sangat penting.
2. Tujuan Perancangan Arsitektur
Arsitektur sistem dirancang untuk:
a. Memisahkan Tanggung Jawab
Agar:
UI tidak bercampur dengan logika ML,
database tidak bercampur dengan tampilan,
preprocessing tidak bercampur dengan controller.
b. Mempermudah Maintenance
Jika terjadi perubahan:
model ML baru,
perubahan threshold,
perubahan database,
maka bagian lain tidak perlu diubah secara besar-besaran.
c. Menjaga Performa Aplikasi
Karena aplikasi melakukan proses berat seperti:
image processing,
tensor conversion,
inference,
maka sistem harus tetap responsif.
3. Penerapan Prinsip SOLID
Sistem menerapkan prinsip SOLID agar struktur kode lebih modular dan maintainable.
4. Single Responsibility Principle (SRP)
Konsep
Satu class hanya memiliki satu tanggung jawab utama.
Implementasi
Contoh Pemisahan




















Dampak Positif
Kode lebih rapi.
Bug lebih mudah dilacak.
Pengembangan fitur lebih cepat.
5. Open/Closed Principle (OCP)
Konsep
Class terbuka untuk pengembangan tetapi tertutup untuk modifikasi langsung.
Implementasi
Model ML dapat diganti tanpa mengubah keseluruhan sistem.
Contoh:
Awalnya menggunakan CNN.
Kemudian ingin mengganti ke MobileNet.
Cukup mengganti implementasi service inference.
Contoh Struktur
abstract class InferenceEngine
 ├── CNNInference
 └── MobileNetInference
6. Liskov Substitution Principle (LSP)
Konsep
Subclass harus dapat menggantikan parent class tanpa merusak sistem.
Implementasi
Semua engine inference memiliki:
method predict(),
input tensor,
output similarity score.
Sehingga controller tidak peduli model apa yang digunakan.
7. Interface Segregation Principle (ISP)
Konsep
Class tidak dipaksa menggunakan interface yang tidak dibutuhkan.
Implementasi
Dipisahkan:
ImageProcessor
DatabaseWriter
FeedbackHandler
Agar setiap service hanya memakai fungsi yang relevan.
8. Dependency Inversion Principle (DIP)
Konsep
Module tingkat tinggi tidak bergantung langsung pada implementasi rendah.
Implementasi
Controller tidak langsung memanggil Hive atau MongoDB.
Tetapi menggunakan abstraction seperti:
StorageRepository
Keuntungan
Jika database berubah:
Hive → SQLite,
MongoDB → Firebase,
controller tetap sama.
9. Single Source of Truth (SSOT)
Konsep
Semua parameter penting disimpan di satu sumber konfigurasi.
Tujuannya:
konsistensi,
kemudahan konfigurasi,
mengurangi hardcoded value.
10. Penggunaan File .env

Contoh:
SIMILARITY_THRESHOLD=0.85 MODEL_INPUT_SIZE=320 PIPELINE_MODE=cnn_v1 
ENABLE_SOUND=true 
ENABLE_VIBRATION=true

aman,
